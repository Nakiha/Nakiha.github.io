储存总览

Android提供了多种自带的储存选项来保存永久性应用数据

  • SharedPreferences - 共享首选项

    提供一个用于保存和检索原始数据的通用框架

  • 内部储存

    直接在设备的内部储存中保存文件, 应用私有的, 其他应用和用户不能访问这些文件

SharedPreference - 共享首选项

这个类提供了一个框架用于保存非结构化的任何原始数据, key-value 模式, 这些数据会跨越多个用户会话永久保留

注意!: 这个类提供了强度很高的一致性保证, 这样的开销在是很昂贵的, 所以如果能容忍数据的丢失, 或者需要高频率地更改数据, 应该使用其他机制

接口描述与从属

可保存的数据类型

  • 布尔值 boolean
  • 浮点值 double
  • 整型值 int
  • 长整型 long
  • 字符串 String

如何使用

关于写入

  1. 获取对象

    有两个方法获取SharedPreference对象

    • getSharedPreferences()

      获取使用键值对来储存的普通SharedPreference 对象

    • getPreferences()

      获取仅用于当前Activity的唯一首选项文件

      这个方法调用了ContextWrapper.getSharedPreferences(java.lang.String, int) , 并且将当前Activitiy的类名作为第一个传参

  2. 写入值

    获取到对象后通过三个步骤来写入值

    1. 调用edit()获取SharedPreferences.Editor
    2. 使用对应的put方法来添加键值对
    3. 使用commit()来提交新值

关于读取

调用对应的get方法, 传入键 并提供默认返回值 来获取原值

内部储存

内部储存的使用围绕着java.io.FileOutputStream和java.io.FileInputStream进行, 进行字节流输入输出

Activity

Activitiy是 Android app 中最重要的一个组件, Activity如何被启动和如何被组合也是 Android app 模型中最基本的一部分, Android 系统将在应用的特定生命期阶段, 调用特定的回调方法. Android 中的回调方法全部以on开头

每个 app 都以数个Activity松散地组合, 这种设计模式使得每个Activity都可以作为被其他应用调用的入口

这使得 Android app 既可以给予用户一个完整的应用体验, 又可以被单独地抽取一些页面作为其他应用的工具来使用

生命周期

onCreate() -> onStart() - > onResume() -> onPause() -> onStop() -> onDestroy()

至于具体的回调顺序, google官方有非常棒的图示, 我这里直接引用官方原图

下面时对各个回调方法进行的简单描述

onCreate() -> onDestroy() 整个生命期

  • onCreate()

    在 Activity 首次被创建时调用

    应该在这里 执行全局状态设置

    例如: 定义与绑定布局, 创建必要的网络连接线程,

  • onDestroy()

    在 Activity 被销毁时调用

    应该在这里 释放所有的资源

    例如: 注销之前创建的 “全局” 广播接收器, 停止创建的网络线程

onStart() -> onStop() 可见生命期

onStart()调用完成后, 意味着程序已经准备好与用户交互, 但需要等待进入前台才能开始交互, 这意味着

  • onStart()

    在 Activity 即将对用户可见之前调用

    可以在这里 注册关于监听UI变化的广播接收器

  • onStop()

    在 Activity 对用户不再可见时调用

onResume() -> onPause() 前台生命期

  • onResume()

    在 Activity 即将开始与用户进行交互之前调用

    onCreateOptionsMenu在这之后调用

    应该在这里 开启动画, 打开本活动需要使用的一些设备(如照相机)

  • onPause()

    当系统即将开始继续另一个 Activity 时调用

    注意: 弹出dialog对话框时, onPause()不会被回调

    应该在这里 确认对未保存的持久性数据的更改, 停止动画以及其他可能消耗 CPU 的内容, 这里的操作需要被非常迅速地完成

异常情形下的生命周期

异常情况下 Activity 会被销毁并重新创建

  • 异常情形1: 资源相关的系统配置改变

    例如 屏幕旋转. 如果想避免活动被销毁重建, 可以对清单文件进行如下设置 

如果不进行设置来覆盖默认操作, 在销毁重建时, activity 依次调用

> `onCreate -> onStart  -> onResume->onPause -> 保存状态 onSaveInstanceState -> onStop -> 销毁 onDestroy -> 重新创建 onCreate -> onStart -> 恢复之前保存的状态 onRestoreInstanceState -> onResume`

其中`onRestoreInstanceState`被保证在`onStop`之前被调用, 并留下一个`Bundle`作为`onCreate`和`onRestoreInstanceState`的传参

`view`和`activity`一样都有`onSaveInstanceState`和`onRestoreInstanceState`方法, 都用于之前状态的恢复, 例如: `EditView`会恢复用户输入的数据, `ListView`会恢复滚动的位置

  • 异常情形2: 内存不足导致低优先级的activity被销毁

    activity的优先级从高到低可分为以下三种

    1. 前台activity

      正在与用户进行交互的activity

    2. 可见但非前台 activity

      非前台的activity弹出了一个dialog

    3. 后台activity

      执行了onStop但未执行onDestroy

Service

(作为四大组件之一, 我至今都没有用过, 学的是个什么 Android 啊……)

Service是一个可以在后台执行长时间运行操作, 但不提供用户界面的应用组件

其他组件可以绑定到相应的服务, 甚至进行进程间通信(IPC Inter-Process Communication 不同进程之间通过共享内存区, 或者磁盘上的普通文件传播或交换信息)

服务最常见的作用: 处理网络事务, 播放音乐, 执行文件I/O, 与内容提供程序交互

参考

谷歌官方指南-组件-Service篇

继承结构

使用服务还是线程?

由于服务是一种即使用户未与应用交互也可以在后台运行的组件, 秉承奥卡姆剃刀原理, 应该只在必要时才创建服务

如果需要在主线程外执行工作, 不过仅仅与用户交互时才有需要, 应该创建新线程

注意! 在默认情况下, 服务仍在应用的主线程中运行 如果需要服务实现密集型或阻止性操作 (例如 MP3 播放或联网) , 仍应在服务内创建新线程, 使主线程专注于前台 Activity 与用户的交互

Service 的两种形式

启动

应用组件通过 调用startService()启动服务时, 服务处于启动状态, 一旦启动, 服务可在后台无期限运行, 直到被指派的事务被完成

启动状态下, 服务通常执行的是一些单一操作, 而且不会将结果返回给调用方. 例如: 点击下载链接后, 浏览器在后台下载文件

绑定

应用组件通过 调用bindService()绑定到服务时, 服务处于绑定状态

绑定状态下, 服务提供了一个客户端-服务器接口, 允许组件与服务进行交互, 发送请求与获取结果, 并且允许多个组件同时绑定到一个服务. 值得注意的是, 如果全部的绑定被取消, 服务会立即被销毁

使用清单文件声明服务

与其他组件一样, 服务应该在清单文件中被声明

android:name是唯一必须的属性, 用于指定服务的类名

Google建议, 为保证应用的安全性, 应仅使用显式 Intent 启动或绑定 Service, 不应该为服务添加intent-filter, 通过更改android:exported属性可以有效阻止其他应用启动服务

服务运行流程

服务由另一个组件通过调用startService()方法来启动, 之后服务会调用 onStartCommand()方法

用于启动服务的 Intent 对象应包含待使用服务的所有数据, 服务会在 onStartCommand中接收到这个 Intent

服务启动后通过stopSelf来自动停止工作, 也可以由另外一个组件调用 stopService来停止

举例:
假设某 Activity 需要将一些数据保存到在线数据库中。该 Activity 可以启动一个协同服务，并通过向 startService() 传递一个 Intent，为该服务提供要保存的数据。服务通过 onStartCommand() 接收 Intent，连接到互联网并执行数据库事务。事务完成之后，服务会自行停止运行并随即被销毁。

拓展两个常见的服务类

  • Service

    适用于所有服务的基类。扩展此类时，必须创建一个用于执行所有服务工作的新线程，因为默认情况下，服务将使用应用的主线程，这会降低应用正在运行的所有 Activity 的性能。

  • IntentService

    实际上大多数启动服务都不必同时处理多个请求, IntentService 是日常使用中最好的选择

    Service 的子类，它使用工作线程逐一处理所有启动请求。只需实现 onHandleIntent() 方法即可，该方法会接收每个启动请求的 Intent，以便执行相应后台工作, 处理完请求后, IntentService会自动停止

拓展 IntentService 类

IntentService 执行以下操作

  • 提供了onBind的默认实现, 简单地返回了null
  • 提供了onStartCommand的默认实现, 创建了一个默认的工作线程, 将传递来的Intent依次发送到工作队列, 和onHandleIntent
  • 处理完全部 Intent后自动结束服务

所以需要使用时, 仅仅需要覆写onHandleIntent和提供一个构造器

如果还准备重写IntentService其他的回调方法, 都应该保证超类实现被调用

拓展 Service 类

由于可以自己处理onStartCommand的每个调用, 因此可以同时执行多个请求

谷歌官方提供的示例的作用与拓展IntentService类时完全相同

onStartCommand返回的整形值用于描述系统该如何在服务终止的情况下继续运行服务

  • START_NOT_STICKY

    如果系统在onStartCommand返回后终止服务, 除非有挂起Intent传递, 否则不会重建服务

  • START_STICKY

    如果系统在onStartCommand返回后终止服务, 重建服务, 并调用onStartCommand, 除非有挂起Intent要启动服务, 否则将用空Intent启动服务

  • START_REDELIVER_INTENT

    如果系统在onStartCommand返回后终止服务, 重建服务, 并通过传递给服务的最后一个Intent调用onStartCommand, 挂起的Intent将会被依次传递

启动服务

通过建立Intent(指定需要启动的服务) 传递给startService, Android 系统将会调用对应服务的(onCreate->)onStartCommand方法, 同时将Intent传递

注意: 如果服务未提供绑定, 则使用 startService() 传递的 Intent 是应用组件与服务之间唯一的通信模式. 如果希望服务返回结果，则启动服务的客户端可以为广播创建一个PendingIntent （使用 getBroadcast()），并通过启动服务的 Intent 传递给服务。然后，服务就可以使用广播传递结果。

插入话题 PendingIntent

pendingIntent字面意思上, 它是一个等待被执行的Intent, 这与它的行为表现所吻合

不过实际上, pendingIntent是一个可被系统维护着的, 用来描述原数据如何被取回的记号, 每个pendingIntent在创建时, 都会被传入上下文对象, 即使建立pendingIntent的进程被销毁, pendingIntent描述的 Action 仍是可用的,

这使得PendingIntent一般作为参数传给某个实例, 而该实例完成某个操作后自动执行PendingIntent上的Action, 也可以通过PendingIntent的send函数手动执行, 并可以在send函数中设置OnFinished表示send成功后执行的动作。

停止服务

启动服务需要自己管理生命周期, 通常系统不会去主动回收其内存资源

有两种停止服务的办法

  • 自行停止stopSelf(int)

    stopSelf应保证在处理完最后一个启动请求后再被使用, 如拓展 Service类的例子中, onStartCommand方法里, 向ServiceHandler的msg.arg1中添加了传入的startId, 这个msg将作为Message对象被传入handleMessage, 处理完毕后, 调用stopSelf(msg.arg1)来进行该次处理是否为最后一次的判定, 如果为真则停止服务

  • 手动停止 stopService()

    于其他组件中调用stopService可以停止指定的服务

Broadcasts

参考

谷歌官方指南-组件-Broadcasts篇

Android 的广播系统是发布者-订阅者 (publish-subscribe) 模式的典型例子

每个广播信息都包装在Intent类中, 并且用特定的标识字符串来描述所广播的内容 (例如android.intent.action.AIRPLANE_MODE), 有的Intent中还会捆绑一些额外信息, 用于描述具体的广播内容, 例如飞行模式的广播就包含有一个描述飞行模式是打开还是关闭的 Boolean 值

值得注意的是, onReceive(Context, Intent)是运行在主线程里的. 而BroadcastReceiver的扩展类也仅仅在onReceive时存活, 如果监听器执行时间超过10秒, app 将会被系统判定为 ANR (Application Not Responding). 所以如果需要执行耗时的工作, 应调用 goAsync()) 或者 使用 JobService 等来启动后台服务完成耗时工作. 因为即使普通地创建了子线程, 广播接收器实例在被销毁后, 子线程也仅有一个宿主进程, 此时它被判定为一个空进程 (没有任何活动组件的进程) , 很容易被系统在需要内存时优先销毁, 子线程也会随之被销毁

Oreo (API level 26) 带来的变化

从Oreo开始, 系统开始对静态注册的广播接收器强加限制, 仍有部分隐式广播得到保留, 剩余的部分请参阅谷歌官方提供的隐式广播例外

这样做的目的是为了减少不必要的后台服务, 毕竟 Android app 开发者一直都喜欢申请不必要的权限, 监听一些不必要的广播, 如果还想使用静态注册的广播, 需要使用 JobScheduler

动态注册

动态注册广播监听器有四个步骤

  1. 创建一个实现了onReceive()方法的BroadcastReceiver子类

  2. 创建BroadcastReceiver的实例

  3. 创建IntentFilter实例, 通过调用 registerReceiver(BroadcastReceiver, IntentFilter)来注册监听器

发送广播

Android 提供了三种方法给 app 来发送广播

  • 发送有序广播 sendOrderedBroadcast(Intent, String)

    顺序广播将根据接收器的<android:priority>属性, 依次递交广播信息, 每个接收器都能完全阻止广播的进一步传播. 同优先级的接收器收到广播的次序是随机的

  • 发送标准广播sendBroadcast(Intent)

    标准广播将会发送给目前全部注册完成的监听器

  • 发送本地广播LocalBroadcastManager.sendBroadcast

    这是最有效率的广播发送形式, 如果不需要跨 app 发送广播, 任何时候都应该使用本地广播

如果单单只需要在自己的应用中使用, 则不需要提供程序, 直接使用 SQLite 即可

OSI七层模型

Posted on 2019-04-30

本文参考自 百度百科 和 钛白Logic 的CSDN博客

1.概念

OSI模型 = Open System Interconnection Reference Model
译名为 开放式互联网通信参考模型 一般称作 OSI参考模型

它由 ISO组织提出, 是一个试图使各种计算机, 在世界范围内互联为网络的标准框架

2.OSI七层模型

OSI模型从 硬件 到 应用 依次为:
物理层 -> 数据链路层 -> 网路层 -> 传输层 -> 会话层 -> 表示层 -> 应用层

3.各层基本定义

鉴于用户是从应用层使用 OSI模型
此处从上到下对模型进行概述

3.1 应用层 Application Layer

负责提供为应用软件建立的接口, 设置于另一应用软件的通信

确定进程之间通信的性质以满足用户的需要。

HTTP, HTTPS, FTP, SSH…

3.2 表达层 Persentation Layer

负责实现数据向合适格式的转换, 提供数据对接收者的系统的兼容
同时该层也可提供对数据的 压缩与加密

3.3 会话层 Session Layer

负责在数据传输途中, 建立, 管理和终止表达层 相应的 通信连接
该层中有不同应用程序的 服务请求和响应

3.4 传输层 Transport Layer

负责给数据添加传输表头 (TH), 将数据打成数据包, 表头中含有所使用的协议等发送信息
建立了源端与目的端的连接, 为上层提供可靠透明的传输服务, 同时屏蔽了更下层传输的细节

解决进程间的通信。

TCP, UDP…

3.5 网络层 Network Layer (IP 协议层)

负责决定数据的路径选择和转寄, 添加网络表头 (NH)到数据包, 表头中含有网络数据,例如IP. 可用于形成分组

解决跨网络的主机通信问题。

IP(互联网协议)

3.6 数据链路层 Data Link Layer

负责网络寻址, 差错校验. 添加数据链表头 (DLH) 和 数据链表尾 (DLT)到数据包中, 形成帧

解决相邻主机通信问题。

  • 数据链表头 包含了物理地址和错误侦测及改错的方法
  • 数据链表尾 包含了一串用于指示数据包末端的字符串

3.6.1此层包含两个子层

  • 逻辑链路控制 LLC (Logic Link Control)

  • 介质访问控制 MAC (media access control)

    MAC子层处理 CSMA/CD 算法, 数据出错校验, 成帧等

3.7 物理层 Physical Layer

负责实现信号的具体化, 落实计算机通信的现实存在

透明地传输比特流。

4.通信特点: 对点通信

源端的每一层都与目的端的对等层进行通信
在每层通信过程中, 使用本层协议进行通信
