高效读取数组: CopyOnWriteArrayList

一些系统资源可能只需要加载或者修改很少的次数, 之后会被系统内的所有模块频繁地访问. 简单地为读写都上锁是很没有效率的. 之前我们讨论了读写分离锁, 它通过两把分离的锁和应用锁降级来完成了读与读之间的完全并发性, 保护了数据的线程安全. 而CopyOnWriteArrayList类更进了一步, 它的写入不会阻塞读取操作, 只有写入与写入, 之间需要进行同步等待.

CopyOnWriteArrayList的写入数据的模式有点像String类, 它会对原有数据进行一次自我复制, 对副本进行修改, 完成后将副本替换掉原来的数据.

对于读取代码, 没有任何的同步控制和锁操作,

而写入时, 需要加锁

插入知识: serialVersionUID

SUID 是源类的哈希值. 所以如果类更新, 例如域的改变, SUID 会变化. JDK中的类经常会手动指定 SUID, 如:

SUID 的值将会在序列化中被使用, 序列化会将类转化为 Byte 流, 用于网络等流式传输. 如果序列化之后更改了类, SUID 变化, 这会导致反序列化时抛出异常, 即 SUID 主要用于版本控制.

插入知识: transient 关键字

来自百度百科 transient

Java 通过序列化提供了持久储存对象实例的机制, 而我们可能会遇到仅仅用来缓存的数据成员, 这时候我们可以给这些缓存成员加上transient关键字, 这样这些缓存数据就不会被序列化, 它们可以降低对不必要的储存空间的开销.

数据共享通道: BlockingQueue 接口

一般, 我们希望整个系统是松散耦合的, 如果想要两个线程 A 和 B 间通信, 我们应该提供一个类扮演中转站一样的角色, 来降低耦合度, 这样我们在后续维护中, 即使重构了 B 线程, 也不需要修改 A , 保证系统的平滑过渡,

ArrayBlockingQueue

BlockingQueue很好的解决了服务线程为空的时候的等待 与 有新消息进入队列后的线程唤醒问题. 这里以ArrayBlockingQueue为例, 探究这是如何实现的.

ArrayBlockingQueue的元素会被放置在一个对象数组中.

它提供了两种压入元素的方法

  • offer()

    如果队列已满, 立即返回false, 未满时才会入列.

  • put()

    如果队列已满, 一直等待, 直到有空闲位置.

两种弹出元素的方法

  • poll()

    如果队列为空, 立即返回null

  • take()

    如果队列为空, 进行等待, 直到有可用元素

这意味着调用put()和take()时可能会引起线程阻塞, 这也就对应了类名中的Blocking

ArrayBlockingQueue内部使用ReentrantLock和Condition(在对Condition 接口的介绍中, 提供的官方示例就是一个ArrayBlockingQueue的简单实现) 实现了等待和通知

随机数据结构: SkipList

SkipList(跳表) , 是一种用来快速查找的数据结构, 类似于平衡树, 但是平衡树的插入和删除常常需要进行全局的调整, 也就得为它准备一个全局锁, 这是很低效的. SkipList只需要局部的调整就可以完成插入和删除, 查询的时间复杂度是 O(log n) 和平衡树相同. 正是由于跳表的局部更改的特性, JDK 使用调表来实现并发数据结构中的Map.

跳表的特点是 随机算法 和 多层链表. 下面的图展示了跳表的结构

每上面一层的链表都是下面一层的子集, 一个元素插入哪些层完全是随机的, 这意味着可能会得到一个性能很糟糕的跳表, 但实际工作中, 它的表现非常好.

对跳表的搜索从顶层开始, 如果没能找到元素, 快速定位到符合的区间后再转入下一层, 重复, 直到找到对应元素.显然, 跳表是一种用空间换时间的算法.
