定义进程与线程

Process 进程

进程是计算机程序关于某数据集合上的一次运行活动, 是系统进行资源分配和调度的基本单位, 是操作系统结构的基础.

在早期面向进程设计的计算机结构中, 进程是程序的基本执行实体; 在当代的面向线程设计进程是线程的容器, 程序是指令数据及组织形式的描述, 进程是程序的实体.

在 Windows 中任务管理器看到的一个个实体就是一个个活的进程

Thread 线程

线程是轻量级的进程, 是程序执行的最小单位.

使用多线程而不是多进程进行并发程序的设计, 主要是因为线程切换和调度的成本远远小于进程.

一个进程中会包含多个线程, 通过多个线程的协作来完成整个 “任务”.

线程的生命周期

线程的所有状态都在Thread中的State中枚举定义

这六个状态也就组成了线程的生命周期

线程的基本操作

新建线程

  • start() 会开启一个新线程
  • run() 在当前线程中执行对应线程的代码

给线程添加 “任务”

重载 run()

这里使用了匿名内部类, 继承了Thread, 通过将run重载把任务填进去

但 Java 是单继承的, 这意味着继承是一种很宝贵的资源, 可使用接口来达到同样的效果

使用 Runnable 接口

在构造Thread类时可以将传递一个Runnable接口来定义任务, 从而避免继承的浪费.

停止线程

曾经用于强制停止一个线程的stop()已被废弃, 它会在调用时直接终止线程, 立即释放这个线程所持有的锁, 这经常会导致一致性错误. 所以, 应该由我们自己来决定如何何时停止线程

例如, 可以通过设置与检测标志符来停止一个线程

线程中断

不同于直接是线程停止的stop(), 线程中断会发送一个通知给线程, 如何处理完全由目标线程自行决定

三个与线程中断相关的方法

如果不对中断操作进行定义, 那这个中断不会发生任何作用

在一个线程里, 可以通过检测线程是否被中断, 来完成中断逻辑的定义

插入话题 Thread.sleep() 方法

Thread.sleep()会让一个线程休眠若干时间, 抛出一个interruptedException(中断异常), 这个异常不是运行时异常, 必须捕获并且处理. 如果在休眠时线程被中断, 这个异常就会产生

注意! 在抛出异常后, 它会清除中断标记, 下一次循环时就无法捕获这个中断

所以, 捕获到中断异常后, 需要再次中断自己, 置上中断标记位

等待和通知

Object提供了wait()和notify()方法, 可以使用它们来完成多线程的协作

例如在线程 A 中, 执行到一个对象调用了obj.wait()后, 线程 A 会停止继续运行执行, 进入obj的等待队列, 直到notify()被调用.

为什么只能在 synchronized 中调用

  • wait()

    用于释放锁, 所以它必须先获得锁

  • notify()

    将锁交给等待队列, 让他们竞争锁, 所以也需要先获得锁

挂起和继续执行

被废弃的方法

suspend()会将线程暂停, 但是却不会释放任何锁资源, 其他任何线程都不能访问被它暂用的锁, 直到执行了resume()方法后, 被挂起的线程才会继续, 被阻塞在相关锁上的线程也可以继续运行.

但是如果resume()意外发生在suspend()前, 被挂起的线程将很难有机会再次得到执行(永久挂起), 而且它的锁不会释放, 本身的线程状态还处在Runnable.

这使得资源不仅得不到释放, 还会误导对当前系统状态的判断.

替代的解决方案

suspend()和resume()方完全可以使用wait()和notify()代替.

添加标记变量, 如果被标记为挂起时, 判断标记, 获取监视器, 执行wait()方法, 进入等待队列.
通过调用resume()方法, 将标记清除, 获取监视器, 执行notify()方法, 唤醒等待队列中的线程.

由于这里只有一个线程会出现在对应的等待队列中, 这样就通过wait()和notify()实现了需要的挂起的效果.

等待线程结束和谦让

等待

很多时候, 一个线程会非常依赖另外一个或多个线程的输出, 这时候需要线程等待一些线程的执行. JDK 提供了join()来实现这个功能

join()为一个同步方法, 它用来阻塞调用它的线程, 使它在获得通知前一直在等待队列中。
被等待的线程在退出前会调用自己的notifyAll()方法, join()方法随之结束

这也就是join()的名字由来, join()使调用它的线程, 加入了对应线程的队列中

谦让

volatile 与 Java 内存模型 (JMM)

volatile关键字保证了变量的可见性和有序性.

使用了volatile关键字后, 相当于告诉了虚拟机, 需要保证这个变量的可见性, 这意味着每个线程拿到这个值的时候一定是目前最新的, 但由于很多操作并非原子性的操作, 例如自增 (获取值, +1, 写回缓存, 刷入主存), 这意味着, 它无法保证复合操作的原子性.

例如: 同时启动多个线程对volatile修饰的变量int i = 0进行自增时, 假定保证每个线程的启动完全同步, 它们在start()时获得的i是 0 , 但是即使一个线程先完成了自增操作i++将i改写成了 1 , 如果另外的线程已经完成i++中获取i的值的步骤, 这个i=1可见性也来得太晚了.
这就是一个典型的volatile关键字无法保证复合操作的原子性的例子.

线程组

ThreadGroup(线程组) 用于提供管理多个线程的服务, 当线程的数目比较多, 而且功能明确时, 可以将相同功能的线程放到线程组中进行管理

常用方法

守护线程

Daemon(守护线程) 是一种特殊的线程, 他是系统的守护者, 在后台进行一些系统性的服务. 比如垃圾回收线程, JIT线程 (Just-In-Time Compiler 即时编译) 等.

与Daemon相对应的是用户线程, 他可以被视为系统的工作线程, 完成系统需要完成的业务操作, 如果用户线程全部结束, 那么整个程序实际上已经无事可做. 因此, 当一个 Java 应用中, 只留下Daemon时, Java 虚拟机会自然地退出.

线程优先级

Java 中线程优先级的有效范围为 1~10, 可通过调用thread.setPriority()来进行设置. Java中提供了 1, 5, 10 三个静态标量来简单地设置优先级.

高优先级的线程在资源竞争中胜出的概率更高, 但是这并不是绝对的, 而且在不同平台上表现不同, 无法被精准预测. 因此在要求严格的场合, 需要自己在应用层解决线程调度问题.

线程安全的概念与 synchronized

在讨论 volatile 时, 遇到了多线程写入冲突的问题, 它是由数据过期导致的, 而使用synchronized可以实现线程间的同步, synchronized通过对需要同步的代码加锁, 使得每一次都只有一个线程能够进入同步块, 进而保证线程的安全性.

关键字sychronized主要有三种用法

  • 指定加锁对象: 给指定对象加锁, 进入同步代码前需要获得给定对象的锁.
  • 直接作用于实例方法: 相对于对当前实例加锁, 进入同步代码前, 需要获得当前实例的锁.
  • 直接作用于静态方法: 相当于对当前类加锁, 进入同步代码前, 要获得当前每的锁.
