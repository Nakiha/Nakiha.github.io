Java并发 - (重入锁与部分任务工具)篇3

Posted on 2019-05-10 | Edited on 2019-07-06

本文主要是介绍一系列的JDK并发包中的工具类. 摘录自葛一鸣与郭超的 [Java高并发程序设计].

Object 监视器方法的另一种实现

通过使用ReentrantLock和Condition, 我们可以实现和synchronized, wait()和notify()一样的效果

重入锁: ReentrantLock

相对于synchronized, ReentrantLock需要自己定义同步代码部分, 通过获得重入锁对象, 显式获得锁与释放锁, 完成对资源的同步.

而且正如ReentrantLock的名字一样, “这把锁” 在一个线程中可以重复进入, 但是获取几次就得释放几次

  • unlock次数 > lock次数

    抛出java.lang.IllegalMonitorStateException

  • unlock次数 < lock次数

    锁未得到释放, 其他线程无法进入临界区

如果线程不一定能获取到锁, 在释放锁时, 可以利用isHeldByCurrentThread()来判断对应的锁是否被该线程持有

中断响应

重入锁提供了可中断支持, 如果需要, 则应该调用lockInterruptibly()方法来获得可中断锁

当中断锁在执行中被中断时, 抛出InterruptedException我们也可以开始在catch或者finally中响应中断.

例如: 线程 A 占用了 lock1 , 线程 B 占用了 lock2 , 这时 A 开始申请 lock2 , B 开始申请 lock1 , 这样构成了一个死锁. 我们可以通过设置线程 B 或者 A 的等待时间, 超时后对其中断, 触发中断响应, 解决这个死锁.

限时等待

除了外部通知, 还可以使用tryLock()方法进行一次限时的等待, 超时后会自动放弃申请锁

公平锁

使用ReentrantLock的无参构造得到的重入锁对象, 或者使用synchronized隐式建立的锁, 都是不公平锁. 不公平锁意味着, 如果两个线程在相近的时间申请同一把锁, 获得锁的线程是随机的, 而非绝对地按照先后顺序.

ReentrantLock利用下面这个构造函数可以得到按照先后顺序分配锁的公平锁对象

为了实现公平锁, 系统必须维护一个有序队列, 这需要更高的实现成本. 如果不使用公平锁, 当两个线程同时申请锁资源时, 系统会倾向于把锁分配给之前拥有过这个锁的线程. 这是高效的, 但是有可能造成另一个线程的饥饿.

Condition 接口

Condition对象创建时就需要绑定到锁, 为了获取特定的Lock实例的Condition, 我们需要调用Lock对象的newConditon()方法, 得到实例后, 我们可以进行一系列的等待, 和通知操作.

一个Lock可以对应多个Condition对象, 来实现每个对象具有多个等待集的效果,

重入锁实现了Lock接口, 所以利用Condition中下面这些方法, 我们可以实现与Object.wait()和Object.notify()一样的功能. 他们都只能在lock.lock()~lock.unlock()之间才能被调用

  • 一系列await()

    使当前线程等待, 直到其他线程使用了signal(), signalALL(), 可指定最长等待时间和使等待不被中断.

  • signal()

    唤醒一个等待中的线程, All 则唤醒所有

限制进程数: Semaphore

无论是之前提到的synchronized还是ReentrantLock都只允许资源在一时间只能被一个线程所访问, Semaphore [ˈseməfɔːr] 可以限制线程数目同时访问某一个资源.

Semaphore维护着一个准入线程数字, 在构造它时, 就需要指定这个准入数, 即同时能申请多少个许可. 可以多传递一个boolean参数指示其是否公平

Semaphore的主要方法有

  • acquire()方法

    尝试获取准入许可, 如果无法获得, 则线程进入等待, 直到有一个线程释放或者当前线程被中断

  • tryacquire()方法

    尝试获取许可, 如果无法获得则放弃

  • release()方法

    释放一个许可, 将其归还给信号量

这里提供官方示例
Pool类使用了信号量来控制并维护了对项目池的项的取出与放入. 即一个取出必须要放回的池, 类似于书架

读写分离锁: ReadWriteLock

ReadWriteLock接口允许并行读取, 同时读写和写写操作会造成阻塞

- 读 写
读 非阻塞 阻塞
写 阻塞 阻塞

读写锁会使用一小段代码来序列化所有线程, 实际上它的开销要比一般的互斥锁更大, 应该根据需求来使用它.

释放写锁前可以申请读锁, 释放读锁之后才能申请写锁.

这里提供官方的示例, 该示例应用了锁降级和实现了ReadWriteLock接口的ReentrantReadWriteLock.

CachedData类中有一个处理缓存数据的方法, 而类中的cacheValid用来标识缓存是否可用. 先获取读锁, 再检测缓存是否可用, 如果不可用, 先释放读锁, 再获取写锁, 由于可能会有别的线程在申请读锁前写入缓存, 所以需要再次检测缓存是否可用, 如果不可用就开始写入缓存, 否则回到原来的操作 - 处理缓存. 在知道了缓存可用后, 为了处理当前的缓存, 需要保证在释放写锁后, 使用数据前, 数据不会改变. 所以需要用到锁降级, 即在释放写锁前就申请回读锁, 保证不会有其他线程的写锁在本线程释放写锁后更改数据.

线程阻塞工具类: LockSupport

LockSupport提供了一系列的park()和一个对应的unpark()方法

调用park()会使当前线程挂起, 并处于WATTING状态. unpark()会解除挂起

这个类为每个线程准备了一个许可, 如果许可可用, park()就会返回, 并消费这个许可, 而unpark()会使这个许可可用. 这意味着如果在park()调用前调用unpark(), 下一次park()将会立即返回, 挂起随之解除.
